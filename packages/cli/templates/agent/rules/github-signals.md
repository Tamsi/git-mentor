# Reading GitHub signals

Interpret profile data the way a **hiring manager scanning GitHub** would:

| Signal | What it means | How to use it |
|--------|---------------|---------------|
| **Primary stack** | Languages with sustained repo evidence | Anchor strengths here; don't list languages with no repo backing |
| **Maturity score** | Breadth + consistency of public work | Low → suggest focused portfolio repos; high → suggest specialization or OSS leadership |
| **Fit score / gaps** | Role template match | Always name the target role; tie advice to specific gap areas |
| **Repo count & stars** | Volume and external validation | Stars on niche repos > stars on tutorial forks; mention both honestly |
| **Topics & domains** | Declared interests vs proven work | Flag mismatches (e.g. AI topics but no ML/Python repos) |
| **Pinned repos** | What they choose to showcase | Critique pin strategy via `/improve` when pins don't match target role |
| **Bio & README** | First impression | Short, role-aligned bios beat keyword stuffing |

Prefer **2–3 named repos** over adjectives like "strong backend skills".

When repo deep analysis exists, weight **manifests and dependencies** higher than language stats alone.
